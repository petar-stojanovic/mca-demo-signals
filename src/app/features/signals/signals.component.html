How Angular Detects Changes

Angular has evolved from checking every component on every event to surgically updating only what changed. Here's the journey — and why it ends with Signals.

Default CD

Any event (click, timeout, HTTP) triggers a full tree traversal. Every component is checked top to bottom — even those unrelated to what changed.

Angular skips entire subtrees unless a component is marked dirty. Still traverses the path from root and re-evaluates all expressions in the dirty component.

From Zoneless to Signals

The problem with Zone.js

Zone.js monkey-patches browser APIs — setTimeout, Promise, fetch, event listeners — so Angular is notified whenever async work completes. Useful, but invisible: it runs constantly, triggers full change detection on every event, and adds ~100 kB to your bundle.

Zoneless removes Zone.js entirely. Angular no longer has an automatic trigger — nothing rerenders unless you explicitly tell it something changed.

Signals fill the gap

Without Zone.js, you need an explicit primitive to signal state changes. That's exactly what Signals are: Angular subscribes to them at the template-binding level. When a signal's value changes, Angular knows which specific bindings depend on it and schedules only those for update.

The result: predictable, fast, and debuggable reactivity. No more "why did this rerender?" — the dependency graph is explicit.
